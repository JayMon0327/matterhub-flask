# MatterHub 개발 워크플로우

## 1. 목적

본 문서는 앞으로 리팩터링 및 기능 구현 시 따라야 할 작업 절차를 정의한다.

## 2. 기본 원칙

- 구현은 항상 문서를 기준으로 진행한다.
- 코드 수정 전에는 반드시 작업 범위를 사용자에게 설명하고 승인을 받는다.
- 기능 완료는 "코드 작성"이 아니라 "테스트 통과 및 검증 완료"를 의미한다.
- 구현 중 더 나은 구조가 보이면 바로 방향을 바꾸지 않고 사용자에게 먼저 제안한다.

## 3. 코드 수정 전 승인 절차

코드를 수정하기 전에 아래 내용을 먼저 공유한다.

- 이번에 구현할 기능 또는 리팩터링 범위
- 수정 예정 파일 또는 디렉토리
- 참조할 문서
- 추가할 테스트 위치
- 예상되는 영향 범위

그 다음 사용자 승인 후에만 코드 수정을 시작한다.

예외:

- 사용자가 명시적으로 요청한 문서 작성 또는 문서 수정 작업
- 코드가 아닌 조사, 구조 파악, 실행 환경 확인

## 4. 문서 우선 원칙

- 구현 전에 반드시 관련 문서를 참조한다.
- 구현 중 문서가 부족하거나 애매하면 먼저 문서를 보강한다.
- 문서와 실제 구현이 달라져야 한다면 문서를 먼저 수정하지 않고, 변경 제안을 사용자에게 올린다.
- 변경 승인을 받은 후 문서를 갱신하고 구현한다.

## 5. 테스트 우선 원칙

- 각 기능은 실제 도메인 디렉토리와 1:1로 매칭되는 테스트 디렉토리에 테스트를 둔다.
- 테스트가 없는 기능은 완료로 간주하지 않는다.
- 테스트는 기능 단위로 추가하고, 해당 테스트를 직접 실행하여 결과를 확인한다.
- 테스트 실행 결과를 작업 마감 시 함께 보고한다.

자세한 규칙은 [테스트 전략](./testing-strategy.md)을 따른다.

## 6. 완료 정의

기능 또는 리팩터링 작업은 아래 조건을 모두 만족해야 완료로 본다.

- 관련 문서를 참조하고 벗어나지 않았을 것
- 필요한 코드가 반영되었을 것
- 대응 테스트 코드가 작성되었을 것
- 테스트가 실제로 실행되었을 것
- 테스트 결과가 정상일 것
- 필요 시 문서가 갱신되었을 것

## 7. 구조 변경 제안 규칙

구현 중 아래 상황이 나오면 바로 코드로 밀어붙이지 않는다.

- 초기 기획보다 더 안전하거나 단순한 구조가 발견된 경우
- 테스트 가능성을 높이는 방향이 확인된 경우
- 운영 복구 관점에서 더 나은 방향이 있는 경우
- OS 또는 배포 환경 제약 때문에 원래 설계를 그대로 적용하기 어려운 경우

이 경우 아래 형식으로 사용자에게 먼저 제안한다.

- 현재 문서 기준
- 새로 제안하는 방향
- 변경 이유
- 장점
- 단점 또는 영향 범위

## 9. 작업 단위 권장 방식

- 한 번에 하나의 도메인 또는 하나의 기능만 진행한다.
- 승인 요청도 기능 단위로 나눈다.
- 테스트 작성과 코드 수정은 같은 작업 단위 안에서 끝낸다.
- 기능이 작더라도 테스트와 검증을 생략하지 않는다.
