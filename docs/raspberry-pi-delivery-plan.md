# 라즈베리파이(Ubuntu 24.04 LTS) 납품용 패키징 및 운영 기획서

## 1. 문서 목적

본 문서는 MatterHub 소프트웨어를 라즈베리파이 + Ubuntu 24.04 LTS 환경에 설치하여 고객사에 납품할 때 필요한 배포, 코드 보호, 원격 유지보수, Wi-Fi 설정 방식에 대한 기준안을 정리한다.

핵심 목표는 다음과 같다.

- 고객사 장비에서 원본 소스코드 노출을 최대한 줄일 것
- 현장 방문 없이 원격 유지보수가 가능할 것
- 고객사가 Linux 셸 로그인 없이 Wi-Fi 정보를 변경할 수 있을 것
- 납품 후 운영 구조가 단순하고 복구 가능해야 할 것

## 2. 전제 및 제약사항

- 대상 장비는 라즈베리파이이며 OS는 Ubuntu 24.04 LTS이다.
- 고객사에는 Linux 로그인 권한을 제공하지 않는 것을 기본 전제로 한다.
- 기존 구조는 MQTT 명령 수신 후 셸 스크립트를 실행하여 `git pull` 기반 업데이트를 수행한다.
- 기존 저장소는 Python 소스코드 중심 구조이며, 납품 시 그대로 배포하면 소스 노출 위험이 높다.

중요한 전제는 다음과 같다.

- 고객사 장비를 직접 소유하는 환경에서 코드 보호는 "완전한 비공개"가 아니라 "노출 난이도 상승"의 개념으로 접근해야 한다.
- 따라서 목표는 Git 저장소 및 평문 Python 소스를 장비에 두지 않고, 실행 파일과 제한된 설정 파일만 남기는 구조를 만드는 것이다.

## 3. 핵심 요구사항

### 3.1 원격 유지보수

다음 두 가지 방식 중 하나가 필요하다.

- 원격 업데이트 기능
- SSH 터널링 기반 원격 접속

현재 논의 기준으로는 자동 업데이트보다 SSH 터널링 기반 유지보수 방식이 더 적합하다.

이유는 다음과 같다.

- `git pull` 방식은 납품 장비에 Git 저장소와 소스코드가 남아야 한다.
- 셸 명령 기반 OTA는 보안 위험이 크고 운영 중 장애 시 복구가 어렵다.
- 터널 기반 유지보수는 장비 내부 상태를 직접 확인하면서 패키지 설치, 로그 점검, 서비스 재시작이 가능하다.

### 3.2 Wi-Fi 설정

고객사가 Linux 로그인 없이 Wi-Fi를 변경할 수 있어야 한다.

이를 위해 장비 내부에서 동작하는 로컬 웹 페이지를 제공한다.

웹 페이지의 필수 기능은 다음과 같다.

- 현재 연결된 Wi-Fi 정보 표시
- 주변 SSID 스캔
- SSID/비밀번호 입력 후 연결
- 연결 성공/실패 결과 표시
- 필요 시 AP 모드로 복구 진입

## 4. 권장 배포 아키텍처

### 4.1 배포 단위

권장 배포 단위는 다음과 같다.

- Python 소스 직접 배포: 비권장
- Docker 이미지 배포: 제한적 권장
- 컴파일된 실행파일 + `.deb` 패키지 배포: 권장

최종 권장안:

- Python 애플리케이션을 `Nuitka`로 컴파일
- 결과물을 Ubuntu용 `.deb` 패키지로 구성
- 장비에는 Git 저장소 대신 설치된 패키지만 남김

### 4.2 파일 배치 기준

권장 디렉토리 구조는 다음과 같다.

```text
/opt/matterhub/bin           실행 파일
/etc/matterhub               환경설정 파일
/var/lib/matterhub           상태 데이터, JSON, 내부 저장소
/var/log/matterhub           애플리케이션 로그
/usr/lib/systemd/system      systemd 서비스 파일
```

배치 원칙은 다음과 같다.

- 실행 코드와 설정 파일을 분리한다.
- 고객사별 설정값은 `/etc/matterhub` 아래에 둔다.
- 런타임 데이터는 `/var/lib/matterhub`에 둔다.
- 로그는 journald 또는 `/var/log/matterhub`에 기록한다.

### 4.3 프로세스 관리

납품 장비에서는 `PM2`보다 `systemd` 사용을 권장한다.

이유는 다음과 같다.

- Ubuntu 서버 환경과 자연스럽게 맞물린다.
- 서비스 자동 시작, 재시작, 로그 조회가 단순하다.
- 파일 시스템 접근 제한, 권한 제한, 환경 변수 주입이 쉽다.
- Node.js 런타임에 대한 추가 의존성을 줄일 수 있다.

권장 방향:

- 현재 다중 Python 프로세스를 하나의 셸 스크립트에서 백그라운드로 띄우는 방식은 지양
- 역할별로 systemd 유닛을 분리
- 예: `matterhub-api.service`, `matterhub-mqtt.service`, `matterhub-rule-engine.service`

## 5. 원격 유지보수 전략

### 5.1 1순위 권장안: Reverse SSH Tunnel

가장 권장하는 구조는 MQTT 또는 내부 스케줄을 통해 장비가 지원 서버로 reverse SSH tunnel을 여는 방식이다.

예시 개념:

```bash
ssh -N -R 2222:localhost:22 maint@support.example.com -p 443
```

운영 방식:

1. 장비는 평소 외부에서 직접 접속되지 않는다.
2. 필요 시 MQTT 명령 또는 관리 API를 통해 "지원 모드"를 활성화한다.
3. 장비가 중앙 지원 서버로 reverse tunnel을 연다.
4. 운영자는 지원 서버를 통해 장비 내부에 접속한다.
5. 접속 후 패키지 교체, 로그 점검, 서비스 재시작을 수행한다.
6. 작업 종료 후 터널을 닫는다.

장점:

- 장비 내부 상태 확인이 가능하다.
- 자동 업데이트 실패보다 복구 가능성이 높다.
- 업데이트 중 문제가 생겨도 운영자가 직접 원인 분석이 가능하다.
- 고객사 네트워크에서 inbound 포트 개방이 필요하지 않다.

필수 보안 조건:

- 비밀번호 로그인 금지
- 공개키 인증만 허용
- 유지보수 전용 계정 분리
- 터널 활성화 시간 제한
- 접속 및 작업 로그 기록

### 5.2 2순위 대안: 서명된 패키지 기반 OTA

자동 업데이트가 꼭 필요한 경우에는 `git pull`이 아니라 "버전 지정형 패키지 설치" 방식으로 구현한다.

권장 흐름:

1. 서버가 특정 버전 정보를 MQTT로 전달
2. 장비가 서명된 `.deb` 또는 사설 APT 저장소의 패키지를 조회
3. 패키지 서명 및 무결성 확인
4. 설치 수행
5. systemd 서비스 재시작
6. 성공/실패 상태를 MQTT로 회신

비권장 요소:

- Git 브랜치명을 MQTT로 내려서 임의 체크아웃 수행
- 셸 문자열 조합 방식으로 외부 명령 실행
- 납품 장비 내부에 Git 작업 디렉토리 유지

## 6. Wi-Fi 설정 웹 페이지 설계 방향

### 6.1 기본 원칙

Ubuntu 24.04 LTS에서는 `wpa_supplicant.conf` 직접 편집보다 `NetworkManager` + `nmcli` 사용을 권장한다.

이유는 다음과 같다.

- Ubuntu 기본 네트워크 관리 방식과 더 잘 맞는다.
- 연결 프로파일 관리가 쉽다.
- 스캔, 연결, 삭제, 상태 조회 API를 만들기 쉽다.
- 복구 시나리오 설계가 용이하다.

### 6.2 제공 기능

웹 페이지에서 제공할 기능은 아래와 같다.

- 현재 네트워크 상태 표시
- 주변 Wi-Fi 목록 스캔
- 새 SSID 연결
- 저장된 연결 목록 조회
- 연결 삭제 또는 우선순위 조정
- AP 모드 진입 또는 복구 모드 진입

### 6.3 내부 구현 방식

웹 UI는 로컬 포트에서만 열고, 내부적으로 제한된 명령만 실행해야 한다.

권장 구현 구조:

- 프론트엔드: 단순 로컬 설정 페이지
- 백엔드: Flask 내부 API 또는 별도 설정 서비스
- 실제 네트워크 변경: 제한된 helper를 통해 `nmcli` 호출

예시 명령:

```bash
nmcli device wifi list ifname wlan0
nmcli device wifi connect "SSID" password "PASSWORD" ifname wlan0
nmcli connection show
nmcli connection delete "CONNECTION_NAME"
```

### 6.4 장애 복구 설계

Wi-Fi 변경 기능에는 반드시 복구 경로가 있어야 한다.

권장 복구 전략:

- 연결 변경 후 일정 시간 내 health check 수행
- 응답 확인 실패 시 이전 연결로 롤백
- 또는 장비가 AP 모드로 자동 전환

예시:

- SSID 변경 후 60초 내 서버 연결 실패 시 이전 프로파일 복구
- 또는 `MatterHub-Setup` 같은 AP를 열고 로컬 재설정 페이지 제공

## 7. 현재 저장소 기준 주요 문제점

현행 코드에는 납품용 구조로 전환하기 전에 반드시 정리해야 할 항목이 있다.

### 7.1 `git pull` 기반 업데이트 의존

현재 저장소는 아래와 같은 구조를 포함한다.

- `app.py`의 `/webhook` 엔드포인트에서 인증 검증 없이 `git pull` 수행
- `mqtt_pkg/callbacks.py`에서 MQTT 업데이트 토픽 수신
- `mqtt_pkg/update.py`에서 외부 셸 스크립트 실행
- `update_server.sh`에서 Git pull, stash, reset, PM2 재시작 수행

문제점:

- 납품 장비에 Git 저장소와 소스코드가 그대로 남는다.
- 브랜치 기반 업데이트는 운영 추적성이 낮다.
- 셸 스크립트 기반 갱신은 복구와 검증이 어렵다.
- 보안상 명령 실행 범위가 넓다.

### 7.2 프로세스 실행 방식

현재 `device_config/matterhub.sh`는 여러 Python 프로세스를 백그라운드로 동시에 기동한다.

문제점:

- 프로세스별 상태 관찰이 어렵다.
- 한 프로세스 장애가 전체 상태에 미치는 영향이 불명확하다.
- 서비스별 재시작 정책을 세밀하게 분리하기 어렵다.

### 7.3 Wi-Fi 정보 관리 방식

현재 저장소에는 `device_config/wpa_supplicant.conf`가 포함되어 있다.

문제점:

- 평문 SSID/비밀번호가 저장소에 남을 수 있다.
- Ubuntu 24.04 운영 방식과 직접 맞지 않을 수 있다.
- 현장 설정 변경을 위한 웹 UI 구조와 분리되어 있지 않다.

## 8. 보안 원칙

### 8.1 코드 보호

- Git 저장소 미배포
- 평문 Python 소스 미배포
- 컴파일 산출물만 탑재
- 디버그 심볼 최소화
- 설정 파일과 인증서 권한 최소화

### 8.2 시스템 접근 제한

- 고객사 일반 계정의 셸 로그인 차단
- `sudo` 권한 미제공
- 유지보수 전용 계정 분리
- SSH는 기본 비활성 또는 내부 전용
- reverse tunnel 활성화 시에만 유지보수 가능하도록 설계

### 8.3 서비스 권한 제한

systemd 유닛에 아래 제한을 적용하는 것을 권장한다.

- `NoNewPrivileges=true`
- `ProtectSystem=strict`
- `ProtectHome=true`
- `ReadWritePaths=` 최소화
- 전용 사용자 계정으로 실행

- MQTT 기반 `git pull` 제거

- Nuitka 기반 빌드

- 재부팅 후 자동 기동 검증

- 패키지 업그레이드/롤백 검증

## 11. 최종 권장 결론

현재 프로젝트의 납품용 목표를 만족하기 위한 최적 구조는 다음과 같다.

- 배포 방식: `Nuitka` 컴파일 + `.deb` 패키지
- 서비스 관리: `systemd`
- 원격 유지보수: reverse SSH tunnel 우선
- Wi-Fi 설정: 로컬 웹 UI + `NetworkManager/nmcli`
- 설정 및 데이터: 실행파일과 분리 저장
- 자동 업데이트: 필요 시 서명된 패키지 기반으로만 제한적으로 도입

즉, 납품 장비에서 `Git 저장소 + PM2 + git pull 업데이트` 구조를 유지하는 것은 권장하지 않는다.

가장 현실적인 방향은 다음과 같다.

1. 소스코드를 컴파일 산출물로 전환한다.
2. 실행 구조를 systemd 중심으로 재구성한다.
3. 유지보수는 reverse SSH tunnel을 기본으로 한다.
4. 고객사용 네트워크 설정은 로컬 웹 UI로 분리한다.
